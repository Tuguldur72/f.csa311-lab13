## 1. Юуг AI хийсэн, юуг өөрөө хийсэн?

Энэ бие даалтыг хийх явцад AI болон хүний оролцоог тодорхой хуваарилж ажилласан.
Тус бүр нь тусдаа үүрэг гүйцэтгэсэн бөгөөд "AI бичнэ, хүн дагана" гэсэн аргаас
зайлсхийхийг хичээсэн.

### А хэсэг — Төлөвлөлт (Plan)

**AI хийсэн зүйлс:**
- Node.js, Python, Go гурван технологийн стекийг харьцуулж, давуу болон сул талуудыг
  жагсаасан хүснэгт гаргасан.
- Mermaid диаграмын кодыг системийн архитектурын дагуу автоматаар үүсгэсэн
  (`graph LR` форматаар, Routes → Controllers → Models → MongoDB бүтэцтэй).
- ADR-001 буюу стек сонголтын архитектурын шийдвэрийн баримтын хэлбэр,
  бүтцийг санал болгосон.
- CLAUDE.md файлын агуулга: build commands, конвенц, no-go zones.

**Өөрөө хийсэн зүйлс:**
- Гурван стекийн харьцуулалтаас Node.js + MongoDB-г сонгох эцсийн шийдвэрийг
  гаргасан. Энэ сонголтыг хийхдээ зөвхөн AI-ийн зөвлөмжид тулгуурлаагүй,
  харин өөрийн локал орчинд MongoDB Compass суулгасан байгаа нь хамгийн
  практик хувилбар байсан гэж үзсэн.
- Хавтасны бүтцийг багшийн заасан `partA/, partB/, partC/` загвартай нарийн
  тулгаж, зарим нэмэлт замуудыг (жишээ нь `adr/` дэд хавтас) өөрөө шийдэн тогтоосон.
- Архитектурын диаграммыг шалгаж, системийн бодит логик урсгалтай нийцэж байгаа
  эсэхийг баталгаажуулсан.

### Б хэсэг — Хэрэгжүүлэлт (Build)

**AI хийсэн зүйлс:**
- `Task.js` MongoDB схемийн анхны хувилбарыг (title, description, priority,
  status, dueDate талбаруудтай) үүсгэсэн.
- `app.js` Express серверийн суурь кодыг (dotenv, mongoose холболт, middleware)
  бичсэн.
- `taskController.js` файл дотор бүх CRUD функцүүдийн (createTask, getAllTasks,
  getTask, updateTask, deleteTask) анхны кодыг гаргасан.
- Jest болон Supertest ашиглан 10 unit тестийн бүтцийг (`describe/it` форматаар)
  боловсруулсан.
- `openapi.yaml` файлын OpenAPI 3.0 спецификацийн бүтцийг үүсгэсэн.

**Өөрөө хийсэн зүйлс:**
- `taskRoutes.js` файл дахь `/:id` параметртэй замуудын (GET, PUT, DELETE) дутуу
  байдлыг аль тест, яагаад унасан болохыг дэлгэрэнгүй error output уншиж
  биечлэн оношилсон.
- `.env` файл дотор локал MongoDB-ийн зөв холболтын мөрийг (`mongodb://127.0.0.1:27017/taskDB`)
  туршиж тохируулсан. Atlas-ийн placeholder хаяг ажиллахгүй байсан тул энэ
  шийдвэрийг өөрөө гаргасан.
- `node_modules` хавтсыг санамсаргүйгээр Git индексэд нэмэх асуудлыг
  `git rm -r --cached .` командаар цэвэрлэж, `.gitignore` нэмсэн.
- Тестийн бүтцэд `createdTaskId` хувьсагчийг тестүүдийн хооронд дамжуулахын тулд
  `let createdTaskId` дээд хэсэгт зарлах шаардлагыг биечлэн дүн шинжилгээ хийж
  тогтоосон.
- Бүх commit мессежийг Conventional Commits форматаар өөрөө боловсруулж,

## 2. Hallucination (AI-ийн төөрөгдлийн жишээнүүд)

### Жишээ 1: Маршрутын дутуу байдал — 4 тест унасан шалтгаан

Хамгийн тод Hallucination нь тестийн файл болон маршрутын файл хоорондын нийцгүй
байдал байсан. AI нь `task.test.js` файл дотор `GET /api/tasks/:id`,
`PUT /api/tasks/:id`, `DELETE /api/tasks/:id` замуудыг ашигласан тест бичсэн.
Гэтэл тэр тестийн кодтой нэг үед бичигдсэн `taskRoutes.js` файлд зөвхөн
`router.route('/')` гэсэн үндсэн зам л тодорхойлогдсон байв.

Үр дүн нь `npm test` ажиллуулахад нийт 10 тестээс 4 нь `Received: 404` алдаатай
унасан явдал байлаа. Ялангуяа тест дугаар 4, 6, 7, 9 буюу ID-аар нэг ажлыг харах,
засах, төлөв өөрчлөх, устгах тестүүд бүгд 404 алдаагаар унасан.

**Яаж олж зассан:** Error output-ыг уншаад `Expected: 200, Received: 404` гэсэн
утгаас зам олдохгүй байгааг дүгнэж, `taskRoutes.js` файлыг нарийн шалгасан.
`router.route('/:id').get(getTask).put(updateTask).delete(deleteTask)` мөр
огт байхгүй байсан нь тодорхой болсон. Энэ мөрийг нэмснийхээ дараа дахин
`npm test` ажиллуулахад бүх 10 тест PASS болсон.

### Жишээ 2: `node_modules` болон Git индексийн асуудал

AI анх `git add .` командыг зааварчлахдаа `.gitignore` файл байгаа эсэхийг
шалгахгүйгээр шууд нэмэхийг санал болгосон. Үүний улмаас `partB/node_modules/`
дотор байгаа олон зуун файл Git индексэд нэмэгдэн, `LF will be replaced by CRLF`
гэсэн мянга мянган анхааруулга дэлгэцэнд гарав.

Хэрэв энэ байдлаар `push` хийсэн бол GitHub repository нь хэдэн зуун MB хэмжээтэй
болж, тестийн `node_modules` файлуудыг бусад хүн татаж авах утгагүй нөхцөл байдал
үүсэх байсан.

**Яаж олж зассан:** `git rm -r --cached .` командаар Git-ийн кэшийг бүхэлд нь
цэвэрлэж, `node_modules/` болон `.env`-г орсон `.gitignore` файл үүсгэсний дараа
дахин `git add .` хийгдсэн. Ингэснээр зөвхөн эх код л индексэд орсон.

## 3. Аюулгүй байдлын эрсдэл — Security Risk

### Mass Assignment (Масс оноолт) эрсдэл

AI-ийн бичсэн анхны `createTask` функц дараах байдалтай байсан:

```javascript
exports.createTask = async (req, res) => {
  const newTask = await Task.create(req.body);
  res.status(201).json({ success: true, data: newTask });
};
```

Энэ код нь `req.body`-г хязгаарлалтгүйгээр шууд `Task.create()` руу дамжуулж байна.
Хэрэглэгч хүсэлтийн body дотор `{ "title": "Ажил", "isAdmin": true }` гэсэн
нэмэлт талбар илгээвэл, Mongoose схемд тэр талбар байхгүй ч бага зэрэг
нөхцөлд шүүлтгүй орж болох эрсдэл бий. Илүү ноцтойгоор хэлвэл, `__proto__`
буюу JavaScript объектийн дотоод талбарыг удирдан нөлөөлөх *prototype pollution*
халдлагын суурь болох боломжтой.

OWASP Top 10 жагсаалтын A03 (Injection) болон A04 (Insecure Design) категорид
орох энэ эрсдэлийг `/security` slash command ашиглан хийсэн аудитаар илрүүлсэн.

**Яаж зассан:** Зөвхөн шаардлагатай талбаруудыг шүүж авах (Whitelist) аргыг хэрэглэсэн:

```javascript
const { title, description, priority, dueDate } = req.body;
if (!title) {
  return res.status(400).json({ success: false, error: 'Гарчиг заавал байх ёстой' });
}
const newTask = await Task.create({ title, description, priority, dueDate });
```

Нэмж, `express-rate-limit` болон `helmet` сангуудыг суулгаж, API-д хурдны хязгаар
(15 минутанд 100 хүсэлт) болон HTTP header-ийн аюулгүй байдлыг тохируулсан.


4. AI-ийн давуу тал (Production Benefit)
Boilerplate код бичих (Express setup, Mongoose Schema) болон OpenAPI спецификаци гаргахад маш их цаг хэмнэсэн.

5. AI-ийн бэрхшээл (Antipatterns)
Логик алдааг оношлохдоо AI заримдаа хэт ерөнхий хариулт өгч байсан. Ялангуяа Git-ийн LF/CRLF анхааруулгыг засахдаа node_modules-ийг индексжүүлж болохгүй гэдгийг анхааруулаагүй нь хүндрэл учруулсан.

6. Skill Atrophy (Ур чадвараа хамгаалах)
Код бичих болгонд AI-аас асуулгүйгээр, эхлээд өөрөө логикоо бодож, дараа нь AI-аар шалгуулах (Review) аргыг хэрэглэсэн. Тестийн алдааг засахдаа AI-аас хариулт авалгүйгээр өөрөө Debug хийсэн.
